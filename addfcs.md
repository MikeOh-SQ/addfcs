# ADDFCS.COM 상세 문서

## 1. 프로젝트 개요

ADDFCS.COM은 주의, 집중, 실행 기능과 관련된 일상 패턴을 사용자가 직접 이해할 수 있도록 돕는 모바일 웹 앱 프로토타입이다. 이 서비스는 의학적 진단이나 선별검사의 대체물이 아니라, 자기보고 설문, 세부 자기점검, 짧은 반응성 과제를 결합해 현재의 주관적 어려움과 수행 기반 반응 패턴을 함께 보여주는 교육 및 연구형 프로토타입이다.

이 프로젝트는 연세대학교 심리과학이노베이션 대학원 `생성형AI의 이해` 과목의 일환으로 진행되었다.

참여자는 김민서, 김수현, 신효주, 오경민, 임수연, 전유민, 최예림이다.

## 2. 서비스 목표

서비스의 핵심 목표는 사용자가 스스로 느끼는 주의·집중 어려움과 짧은 수행 과제에서 나타나는 반응 패턴을 한 화면에서 비교할 수 있게 하는 것이다. 사용자는 단순히 높은 점수나 낮은 점수를 받는 것이 아니라, 어떤 영역에서 부담이 나타나는지, 자기보고와 수행 지표가 비슷한 방향인지, 그 결과를 바탕으로 어떤 실천 계획을 세울 수 있는지 확인한다.

서비스는 다음 원칙을 따른다.

- 진단보다 자기이해를 우선한다.
- AI가 단독으로 결론을 내리지 않고 저장된 응답과 계산값을 우선한다.
- 반응성 과제는 보조 자료로 사용하며, 과제 하나로 ADHD 여부를 판단하지 않는다.
- 결과는 부주의, 충동성, 활동성, 일상 피로도, 실행 계획의 형태로 쉽게 설명한다.
- 사용자의 기록은 `database/*.json`에 저장되고, GitHub 백업에서는 현재 제외 규칙에 따라 제외된다.

## 3. 전체 사용자 흐름

서비스는 단일 페이지 앱 구조로 진행된다.

1. `ID`: 새 ID를 만들거나 기존 기록을 불러온다.
2. `SELF`: 6문항 간단 자기점검을 수행한다.
3. `SYM`: 23문항 세부 자기점검을 수행한다.
4. `REACT`: 반응성 과제 3종을 수행한다.
5. `RESULT`: 저장된 응답과 과제 결과를 바탕으로 패턴 리포트를 확인한다.
6. `PLAN`: 유형별 실행 계획과 Gemini 기반 조정 채팅을 확인한다.
7. `DTx`와 `plangame`: 계획 실천과 미니게임 점수 누적을 통해 후속 행동 루프를 제공한다.
8. `/map`: 서비스의 JSON, AI 입력, 화면 흐름, fallback 구조를 시각화한다.
9. `/analysis`: 저장 기록을 바탕으로 연구 분석용 요약 지표를 확인한다.

하단 네비게이션은 현재 단계 표시용이며, 사용자가 임의로 단계를 건너뛰는 흐름은 제한되어 있다. 특정 개발·검증 상황에서는 URL query shortcut으로 특정 단계에 진입할 수 있다.

## 5. 자기점검 구성

### 5.1 간단 자기점검

`config/asrs.json`에는 최근 6개월 기준의 6문항 자기점검이 들어 있다. 각 문항은 0~4 척도다.

- 0: 전혀 그렇지 않다
- 1: 드물게 그렇다
- 2: 가끔 그렇다
- 3: 자주 그렇다
- 4: 매우 자주 그렇다

문항은 세부 실수, 집중 유지, 듣기 어려움, 미루기, 가만히 있기 어려움, 말 끊기와 같은 일상 장면을 다룬다. 앱은 이 응답을 통해 집중 유지 부담과 반응 조절 부담을 계산하고, 이후 리포트의 주관적 보고 영역에 반영한다.

### 5.2 세부 자기점검

`config/dsm-5.json`에는 23문항 세부 자기점검이 들어 있다. 문항은 `부주의`, `과잉행동/충동성`, `추가 확인`으로 나뉜다.

- 부주의: 9문항
- 과잉행동/충동성: 9문항
- 추가 확인: 5문항

추가 확인 문항은 지속 기간, 아동기부터의 양상, 여러 환경에서의 반복, 기능 저하, 다른 요인만으로 설명되는지 여부를 묻는다. 이 문항들은 진단을 내리기 위한 용도가 아니라 자기이해 리포트에서 맥락을 풍부하게 하는 보조 정보로 사용된다.

## 6. 반응성 과제 구성

반응성 과제는 짧은 수행 기반 보조 지표를 제공한다. 이 지표는 설문과 함께 해석되며, 과제 결과만으로 과잉행동, 충동성, 부주의를 단정하지 않는다.

### 6.1 신호 찾기

신호 찾기는 목표 자극에만 반응하고 비목표 자극에는 반응하지 않는 과제다.

- 본 시행: 60회
- 자극 시간: 500ms
- ISI: 1000, 1250, 1500ms 중 무작위
- 주요 지표: `omission_rate`, `reaction_time_variability`, `tau`, `late_phase_drop`
- 해석 초점: 목표 놓침, 반응시간의 일관성, 후반부 집중 유지

### 6.2 멈춤 버튼

멈춤 버튼은 Go/No-Go 방식의 반응 억제 과제다.

- 본 시행: 50회
- Go 자극: `o.gif`
- No-Go 자극: `x.gif`
- 자극 시간: 500ms
- ISI: 800, 1000, 1200ms 중 무작위
- 성공 기준: Go 자극 시작 후 220ms 이후부터 자극 종료 전까지의 반응
- Apple touch device 보정: 터치와 시각 지연을 고려해 380ms 보정 적용
- 주요 지표: `commission_rate`, `fast_error_rate`, `mean_go_reaction_time`, `premature_response_count`
- 해석 초점: 성급 반응, 잘못된 반응, 멈추는 조절

### 6.3 균형 유지

균형 유지는 활동성 및 자기조절과 관련된 보조 지표를 제공한다.

- 수행 시간: 30초
- 입력 방식: 가능한 경우 모바일 센서, 불가능한 경우 중앙 원 롱터치 또는 포인터 fallback
- 주요 지표: `stable_duration_pct`, `spike_count`, `total_movement`, `input_source`
- 해석 초점: 안정 유지 시간, 큰 흔들림, 움직임 조절

## 7. 리포트 생성 방식

이 서비스의 리포트는 AI가 임의로 작성하는 문서가 아니다. 핵심 구조는 결정론적 계산을 우선하고, AI는 표현과 계획 조정을 보조한다.

서버는 저장된 record를 읽어 통합 metrics를 계산한다. 주요 계산 재료는 다음과 같다.

- 간단 자기점검 응답
- 세부 자기점검 Yes/No 응답
- 반응성 과제 raw 지표
- 과제 완료 여부
- 자기보고 영역과 객관 수행 영역의 상대적 방향
- 일상 피로도 수준

`buildDeterministicReport()`는 다음 구조의 리포트를 만든다.

- `severity`: 낮음, 중간, 높음
- `scores`: attention, executive, impulse, emotion, structure
- `hero`: 핵심 뱃지와 요약
- `crossCheck`: 자기보고와 수행 과제의 일치·불일치
- `burdenPattern`: 주관 부담과 수행 부담의 조합
- `profile`: 부주의, 충동성, 활동성 설명
- `dailyImpact`: 일상 피로도
- `bridge`: 계획 화면으로 이어지는 문구

## 8. 부담 패턴과 해석 원칙

리포트는 단일 점수보다 패턴을 강조한다. 특히 주관적 부담과 수행 기반 부담을 함께 비교한다.

- 주관적 보고가 높고 수행 지표도 부담을 보이면 사용자가 느끼는 어려움과 과제 반응이 같은 방향으로 나타난 것으로 해석한다.
- 주관적 보고는 높지만 수행 지표가 안정적이면 특정 상황, 피로, 환경, 긴장도 같은 맥락 차이를 고려한다.
- 주관적 보고는 낮지만 수행 지표에서 부담이 보이면 사용자가 일상에서 잘 인식하지 못한 수행 패턴이 있을 수 있음을 조심스럽게 설명한다.
- 두 지표가 모두 안정적이면 현재 기록에서는 큰 부담이 높게 나타나지 않는다고 안내하되, 수면과 스트레스 변화에 따라 체감은 달라질 수 있음을 덧붙인다.

## 9. AI 활용 방식

Gemini는 다음 API에서 사용된다.

- `POST /api/ai/asrs-analysis`
- `POST /api/ai/dsm-analysis`
- `POST /api/ai/react-analysis`
- `POST /api/ai/insights`
- `POST /api/ai/chat`

AI 입력에는 RAG 검색 결과가 아니라 현재 record, local analysis, metrics가 직접 들어간다. Gemini가 설정되지 않았거나 응답이 불완전한 경우 서버와 프런트는 로컬 분석 및 결정론적 리포트를 fallback으로 사용한다.

`POST /api/ai/insights`는 리포트와 계획을 생성하지만, 최종 리포트의 핵심 점수와 부담 패턴은 서버 계산값을 유지한다. AI가 바꿀 수 있는 영역은 주로 문장 표현, 요약, 실행 계획의 자연스러운 설명이다.

`POST /api/ai/chat`은 계획 화면에서 사용자의 생활 맥락에 맞춰 실행 계획을 조정하는 용도로 사용된다.

## 10. 실행 계획 생성

`buildPlanForMetrics()`는 사용자 패턴에 따라 기본 실행 계획 3개를 만든다.

- 부주의형 경향: 과제 분할, 시각적 단서, 환경 통제, 생각 노트
- 충동성 경향: 심호흡, 메모, 감정과 행동 기록
- 복합형 경향: 작은 행동 단위, 충동 지연, 하루 마무리 점검
- 안정 범위: 루틴 유지, 집중 시간대 기록, 반복 가능한 생활 습관

계획 화면에서는 이 기본 제안을 보여주고, Gemini 채팅을 통해 현실 생활에 맞게 조정할 수 있다. 이후 `plangame`은 이 계획을 목표 카드로 사용한다.

## 11. DTx와 plangame

`/dtx`와 `/plangame`은 평가 이후의 행동 루프를 담당한다. 새 리포트를 다시 만드는 화면이 아니라, 이미 저장된 `plan`, `record.dtx`, `record.planGame`, `record.tutorials` 상태를 사용한다.

- `DTx`: 숲 허브, 단계 이미지, 누적 점수, 튜토리얼 대사
- `plangame`: 계획 목표 카드, 목표 수정, 체크 상태
- `game/scripts/*.json`: 튜토리얼과 대사 데이터
- `game/images/*.png`: 캐릭터와 단계 이미지

## 12. 데이터 저장 구조

사용자 기록은 `database/<id>-<timestamp>.json` 형태로 저장된다. 주요 필드는 다음과 같다.

- `id`: 사용자 ID
- `createdAt`, `updatedAt`: 생성 및 수정 시각
- `currentStep`: 현재 단계
- `asrs`: 간단 자기점검 응답
- `dsm5`: 세부 자기점검 응답
- `game.tests`: 반응성 과제별 결과
- `reactivityAnalysis`: 반응성 과제 해석
- `report`: 통합 리포트
- `plan`: 실행 계획과 채팅
- `dtx`: DTx 점수와 단계
- `planGame`: 계획 게임 상태
- `researchUsage`: 연구 분석용 세션 및 활동 이벤트

서버의 `saveRecord()`는 저장 과정에서 기존 DTx, planGame, tutorials 상태가 의도치 않게 사라지지 않도록 일부 필드를 보존한다.

## 13. API 목록

주요 API는 다음과 같다.

- `GET /api/config/:file`: 설문 및 리포트 설정 조회
- `GET /api/records`: 저장 기록 목록 조회
- `GET /api/records/:fileName`: 개별 기록 조회
- `POST /api/records`: 기록 저장
- `GET /api/analysis`: 연구 분석용 요약 조회
- `GET /api/design-themes`: 디자인 테마 목록 조회
- `GET /api/design-themes/:slug`: 특정 디자인 테마 조회
- `GET /api/map-layout`: 서비스 지도 노드 위치 조회
- `POST /api/map-layout`: 서비스 지도 노드 위치 저장
- `GET /api/ai/status`: Gemini 설정 여부 조회
- `POST /api/ai/insights`: 통합 리포트와 계획 생성
- `POST /api/ai/chat`: 계획 조정 채팅
- `POST /api/ai/asrs-analysis`: 간단 자기점검 AI 설명
- `POST /api/ai/dsm-analysis`: 세부 자기점검 AI 설명
- `POST /api/ai/react-analysis`: 반응성 과제 AI 설명

## 14. 디자인 테마

`designmd/DESIGN-*.md` 파일은 서버가 자동으로 읽는다. 각 파일의 front matter에는 색상, 타이포그래피, 둥글기, 간격, 컴포넌트 스타일 같은 토큰이 들어 있다. 앱은 `/api/design-themes`와 `/api/design-themes/:slug`로 이 값을 받아 CSS 변수 기반 테마로 변환한다.

## 15. 연구 분석 화면

`/analysis` 화면은 저장된 기록을 바탕으로 연구용 요약 지표를 보여준다. 서버는 `buildAnalysisSummary()`를 통해 완료 단계, 설문 선택, 반응성 과제 완료자, 주요 지표 평균, 상관 분석용 포인트 등을 구성한다. 분석 화면은 서비스 운영 중 기록의 흐름을 살피기 위한 보조 도구다.

## 16. 이론적 배경

서비스의 이론적 배경은 ADHD와 실행 기능, 자기보고와 수행 기반 측정의 차이, 생성형 AI의 보조적 해석 가능성에 있다.

주의와 실행 기능의 어려움은 단일 행동으로 드러나기보다 실수, 지속주의 저하, 과제 시작 지연, 조직화 어려움, 반응 억제 어려움, 정서적 피로 같은 여러 패턴으로 나타날 수 있다. 따라서 서비스는 하나의 점수로 결론을 내리지 않고 주관적 보고와 수행 기반 반응을 나누어 본다.

자기보고 설문은 사용자가 평소 일상에서 느끼는 어려움을 잘 반영하지만, 기억 편향, 현재 기분, 스트레스 수준의 영향을 받을 수 있다. 반응성 과제는 짧은 실험적 상황에서의 반응 패턴을 보여주지만, 실제 생활 전체를 대표하지는 않는다. ADDFCS.COM은 이 두 자료를 함께 보여주어 사용자가 자기 패턴을 더 입체적으로 이해하도록 설계되었다.

생성형 AI는 이 프로젝트에서 판단자가 아니라 설명 보조자다. AI는 사용자의 record와 계산된 metrics를 입력받아 이해하기 쉬운 한국어 문장과 실행 계획 조정안을 만든다. 핵심 점수, 부담 패턴, 리포트 구조는 결정론적 계산이 우선한다.

## 17. 안전과 한계

이 서비스는 의학적 진단 도구가 아니다. 결과는 ADHD 여부를 확정하지 않으며, 병원 진료, 전문가 상담, 표준화된 심리검사를 대체할 수 없다.

다음 한계를 명시한다.

- 짧은 반응성 과제는 사용자의 전체 인지 기능을 대표하지 않는다.
- 모바일 기기, 브라우저, 터치 지연, 센서 권한에 따라 결과가 달라질 수 있다.
- 자기보고는 사용자의 현재 기분, 수면, 스트레스, 회상 방식에 영향을 받을 수 있다.
- AI 문장은 입력된 record와 metrics에 기반하지만, 표현의 자연스러움을 위한 보조 기능일 뿐이다.
- 리포트는 수면 부족, 불안, 우울, 스트레스, 환경 변화 같은 다른 요인을 함께 고려해야 한다.

## 18. 실행 방법

루트에 `.env` 파일을 만들고 필요한 환경 변수를 설정한다.

```env
GEMINI_API_KEY=your_api_key
GEMINI_MODEL=gemini-2.5-flash
HOST=0.0.0.0
PORT=3333
```

실행 명령은 다음과 같다.

```bash
npm install
npm start
```

백그라운드 실행은 `./run.sh`, 서버 중단은 `./end.sh`를 사용한다.

## 19. 유용한 경로

- `/`: 메인 앱
- `/?theme=bmw-m`: 디자인 테마 강제 적용
- `/?route=report&id=test01`: 특정 ID의 리포트 단계 진입
- `/?shortcut=reactivity&test=go_nogo&id=test01`: 반응성 과제 shortcut
- `/?shortcut=reactivity-result&test=balance_hold&id=test01`: 반응성 완료 화면 shortcut
- `/dtx?id=test01`: DTx 숲 허브
- `/plangame?id=test01`: 플랜 게임
- `/map`: 서비스 구조도
- `/analysis`: 연구 분석 화면
